# 综合需求分析报告：PI批量导出功能优化

分析日期：2026-06-30

## 1. 需求概述

本需求是对 BPS 系统中 PI 导出能力的优化：业务希望在"我的 PO"页面一次输入多个 PO 号，由系统自动汇总生成 PI Excel，减少跟单人工复制、合并、校验多个 PO 明细的工作量。

基于会议纪要和 BPS-WS 代码初步判断：需求可行，但不是单纯修改 Excel 模板。系统现有能力以单 PO 查询、单 PO 明细导出、PO 列表导出、PI 模板填充逻辑为主；要实现批量 PI，需要新增或扩展批量入口、PO 分组校验、数据聚合、金额计算、返利/定金/收款账户取数和异常提示逻辑。

## 2. 需求背景与目标（Context）

### 2.1 需求背景

当前业务在处理海外门店建店、展会、批量下单等场景时，经常出现一个客户对应多个 PO、一个批次包含大量 SKU 的情况。现有系统导出后，跟单需要手工将多个 PO 的 PI 内容合并成一个 Excel，并补充原始小 PO 号、核对金额、调整模板字段，单次处理通常需要 2-3 小时。

会议中业务明确提出，希望系统能基于多个 PO 自动生成合并 PI，同时保留原始 PO 的可追溯信息，减少人工合并和金额核对风险。

### 2.2 需求目标

1. 在商城端"我的 PO"页面支持一次输入多个 PO 号，并导出合并后的 PI Excel。
2. 同一客户下的多个 PO 可以合并导出；不同客户的 PO 不应合并到同一份 PI。
3. 导出明细中增加原始小 PO 号字段，便于业务和客户追溯。
4. PI 表头、客户信息、贸易条款、收款账户、金额字段尽量沿用现有单 PO PI 逻辑。
5. 支持业务常用的批量输入方式，例如从 Excel 粘贴多行 PO、使用换行、空格、逗号分隔。
6. 优化导出状态筛选，覆盖业务实际需要导出的 PO 状态。

### 2.3 核心痛点

1. 手工合并耗时长，重复操作多。
2. 多 PO、多 SKU 场景下容易漏 SKU、错 PO、错金额。
3. 原始 PO 与合并 PI 的追溯关系不够清晰。
4. PI 收款账户、返利、定金、运费、保险等字段规则存在人工判断，系统规则需要统一。
5. 商城端和运营端可能存在相似入口，但当前需求边界尚未完全确认。

## 3. 系统可行性初判

**结论：可行，建议按"中等复杂度改造"评估**；一期应优先做商城端批量 PI 导出，不建议同时扩展运营端和邮件发送链路。

### 3.3 主要技术/数据/权限/流程风险

1. **金额公式口径风险（高）**：代码公式和会议公式有结构性差异（保险费的有无、押金字段的有无）。如果公式确认延后，导出金额可能与业务预期不一致。
2. **银行账户数据风险（高）**：如果维持动态查询但账户数据不准确，导出文件可能包含错误账户信息，直接影响客户收款。
3. **返利数据准确性风险（中）**：`contractEarnest`（汇总字段）和 `LsPoRebateDetail`（明细表）可能不一致，导出时应选择哪个来源需要业务确认。
4. **性能风险（中）**：虽然批量查询能力已存在，但大规模 PI 导出（50+ PO、数百 SKU）加上 EasyExcel 模板填充，首次上线前需要进行性能压测。
5. **SQL 查询缺失风险（中）**：`OcContractLSMapper.xml` 未读取到，PO → SKU 明细的完整 SQL 查询逻辑尚未确认，可能存在字段遗漏或 JOIN 链复杂的问题。

### 3.4 最可能的实现路径

**推荐方案 C：新建独立批量 PI 导出接口**

```
用户输入 PO 列表
    → 前端解析 (codeSet 扩展：兼容换行/空格/逗号)
    → 后端校验 (客户/币种/贸易条款一致性)
    → 批量查询 SKU 明细 (复用 getQuantityPoList Feign)
    → 金额计算 (确认公式口径后实现)
    → 表头组装 (处理多 PO 冲突规则)
    → 银行信息获取 (按确认策略)
    → EasyExcel 模板填充 (新增批量版 OSS 模板)
    → 输出 Excel
```

**原因**：
1. `exportUserPoDetail.json` 当前更像"用户 PO 明细导出"，不完整覆盖 PI 表头和金额。
2. `ContractCon.downloadPoExcel()` 更接近目标导出形态，但方法体已将单 PO 逻辑和模板填充深度耦合。
3. 新增接口可减少对现有单 PO 导出的影响，也便于运营端后续复用。

### 3.5 不建议纳入一期的范围

1. 运营端同步改造（建议后续复用后端接口）。
2. 批量发送 PI 邮件（本期只导不出，不发送）。
3. 自动生成新 PI/PU 编号并回写订单（沿用第一张 PO 的编号或文件命名方案）。
4. 账户主数据维护流程改造（一期采用固定账户或读取最新有效账户）。
5. 超大批量异步导出（一期限制 20-50 个 PO，走同步下载）。

### 4.3 金额计算公式（关键发现 - 代码 vs 会议口径不一致）

**代码实际公式**（`ContractCon.java` ~5274-5289）：

```java
BigDecimal total = subTotal.add(freight).add(discount).subtract(insurance).subtract(other);
```

各字段来源：

| 变量 | 系统字段 | 语义 | 值特征 |
|------|------|------|------|
| `subTotal` | `totalPricesetNprice` | SKU 标准总价（数量 × 单价） | 正数 |
| `freight` | `goodsLogmoney` | 运费 | 正数或 0 |
| `discount` | `goodsPmoney` + `goodsPinprice` | 优惠金额 + 改价金额 | **均为负数**（故 `add` 等价于减去绝对值） |
| `insurance` | `contractAmoney` | 保险费 | 正数，代码中 **subtract** |
| `other` | `contractEarnest` | 返利（其他扣款） | 正数或 0，代码中 **subtract** |

**等价代数式**：
```
实际总额 = 标准总价 + 运费 + 优惠(负值) + 改价(负值) - 保险费 - 返利
```

**PoExcel.java 中另一套公式**（~975-985）：
```
inmoney - goodsPmoney + goodsPinprice - contractEarnest + goodsLogmoney
```

**会议口径**：
```
实际总额 = 标准总金额 - 调价 - 返利 + 运费 - 押金
```

**差异对比**：

| 差异点 | 代码行为 | 会议口径 | 影响 |
|------|------|------|------|
| **保险费** | 从总额中**扣除** (`subtract insurance`) | **未提及** | 如果实际总额不应扣除保险费，会议口径与代码口径存在结构性冲突 |
| **押金/定金** | 代码中**不存在** `deposit` 字段 | 公式中包含 `- 押金` | 需要新增字段并在公式中调整 |
| **优惠和改价符号** | 数据库存储为负数，代码用 `add`（即加负数=减） | 统一称为"调价" | 业务认知与系统存储可能不一致，需确认正负号规则 |
| **返利字段** | `contractEarnest`，单字段 | 明确要求返利字段 | 代码已有，但数据来源和准确性需确认 |

→ **结论**：金额计算**不能**简单复用现有代码公式。开发前必须与业务确认：
  1. 保险费是否应计入总额（代码扣除了，会议未提）。
  2. 押金/定金字段的来源和计算规则（会议要求，代码完全缺失）。
  3. 优惠、改价的正负号展示规则（数据库存负数，模板展示应取绝对值还是原值）。

### 4.5 银行账户信息（关键发现 - 代码动态查询 vs 会议固定账户）

**代码行为**（`ContractCon.java` ~5385-5408）：
```java
// 通过 ocContractReDomain 获取 PO 关联的客户银行信息
String accountName = ocContractReDomain.getFaccountName();  // 收款账户名
// 进一步调用 cp.userbank.queryUserbankPage 分页查询客户银行信息
```

**银行账户信息模型** `CpUserbankDomain` 字段：
- 账户名称、账号、开户行、Swift Code、银行地址、公司抬头
- 每个 PO 关联的客户可能有**多套**银行信息

**会议口径**：固定展示**香港收款账户**，不受 PO 原账户影响。

**关键冲突**：
- 代码目前是**动态查询**每个 PO 关联客户的银行信息，意味着不同 PO 可能展示不同账户。
- 会议明确要求统一为香港账户（固定值或财务维护的默认账户）。
- 如果采用固定账户方案，现有 `getBankInfo()` 逻辑需要改造，且涉及账户数据源的权限责任确认（会议提到财务中心为最终责任人）。

### 4.7 PI 导出 vs PO 明细导出（两者不同，需选择）

**PI 导出**（ContractCon.java `downloadPoExcel()`）：
- 模板：PI/PU 格式，含表头（客户信息、贸易条款、银行账户）、明细（SKU 行）、金额汇总
- 字段数：28 个（含 SKU 编码、品名、规格、数量、单价、金额、箱规、毛净重等完整 PI 字段）
- 金额计算：有完整的总额计算逻辑

**用户 PO 明细导出**（ContractCon.java `exportUserPoDetail.json`）：
- 模板：简约明细格式，仅 12 个字段（SKU 编码、规格描述、数量、单价、客户 SKU 等）
- 不含：表头客户信息、贸易条款、银行账户、金额汇总、返利、运费
- 更偏"订单明细查看"，不适用于 PI 场景

**对需求的影响**：
- 业务目标明确为"PI 导出"，因此应**基于** `downloadPoExcel()` 逻辑扩展，而非 `exportUserPoDetail.json`。
- 但 `exportUserPoDetail.json` 的**批量输入模式**可参考（统一入口、多 PO 输入、状态筛选）。

### 4.8 多 PO 输入解析参考

`ContractCon.java` 中已有 `codeSet()` 方法（~5110），实现逻辑：
```java
private Set<String> codeSet(String codes) {
    Set<String> set = new HashSet<>();
    if (StringUtils.hasText(codes)) {
        // 按逗号分隔后 trim 去重
        String[] arr = codes.split(",");
        for (String s : arr) {
            if (StringUtils.hasText(s)) {
                set.add(s.trim());
            }
        }
    }
    return set;
}
```

→ 可以直接复用或扩展该方法，支持换行、空格、逗号、中文逗号分隔。

## 5. 待澄清核心问题

### 1.[ 系统逻辑 ] 产品经理梳理：

1. 本次应基于哪条现有导出链路改造？
   - [A] 基于商城端 `exportUserPoDetail.json` 扩展为多 PO 导出，改造成本低，但该链路当前更偏 PO 明细，不完整覆盖 PI 表头和金额。
   - [B] 基于 `ContractCon` 中现有 PI 模板填充逻辑扩展批量能力，更贴近 PI，但需要新增多 PO 聚合和冲突处理。
   - [C] 新建独立批量 PI 导出接口，复用现有查询、模板、金额和银行逻辑，对旧功能影响最小。

2. 用户输入的"PO号"在系统中应对应哪个字段？
   - [A] 对应 `CONTRACT_NBILLCODE`，即当前 `getUserPoDetail` 使用的外部/用户 PO 号。
   - [B] 对应 `CONTRACT_BILLCODE`，即系统内部 BPS/合同单号。
   - [C] 同时支持两类编号，系统自动识别，但需要明确冲突时优先级和提示文案。

3. 多 PO 合并的客户一致性应按哪个字段判断？
   - [A] 按 `MEMBER_BCODE` 判断，同一客户编码才允许合并。
   - [B] 按客户名称判断，名称一致即可合并，但存在同名或改名风险。
   - [C] 按客户编码、币种、贸易条款同时一致才允许合并，规则更严格但失败率更高。

4. 如果输入的 PO 属于多个客户，系统输出形式如何设计？
   - [A] 直接阻断导出，提示用户只能选择同一客户 PO。

   - [C] 按客户拆成同一个 Excel 的多个 Sheet，但客户转发和归档时可能不方便。

5. 多个 PO 中相同 SKU 是否合并为一行？
   - [A] 不合并，按 PO + SKU 保留原始行，追溯最清晰。
   - [B] 同客户、同 SKU、同价格、同规格合并数量，表格更短但追溯较弱。
   - [C] 默认不合并，同时增加汇总区域展示同 SKU 合计。

6. 原始小 PO 号列应展示哪个系统字段？
   - [A] 展示 `CONTRACT_NBILLCODE`，与当前用户 PO 查询逻辑一致。
   - [B] 展示 `CONTRACT_USERURL` 或现有 PI/客户 PO 字段，需确认业务实际认知。
   - [C] 同时展示系统 PO 号和客户原始 PO 号，字段更完整但模板更宽。

7. 多 PO 合并后 PI 编号应如何生成？
   - [A] 沿用第一张 PO 的 PI/PU 编号。
   - [B] 生成新的批量 PI 编号，并维护与原 PO 的关联。
   - [C] 不生成新编号，仅在文件名和表内备注展示多个 PO 号。

8. 表头字段冲突时如何处理？
   - [A] 以第一张 PO 为准，导出后由业务人工检查。

   - [C] 允许部分字段不一致，在表头展示"多值"或备注，明细保留各 PO 信息。

9. 币种不一致时是否允许合并？

## 6. 建议的一期边界

建议一期范围：

1. 入口：商城端"我的 PO"新增批量 PI 导出。
2. 输入：支持粘贴多行、空格、逗号分隔 PO，并自动去重。
3. 校验：按客户编码、币种、贸易条款做一致性校验；不一致时提示拆分或阻断。
4. 输出：同客户生成一份 Excel；跨客户建议打包为多份 Excel 或先阻断，需业务确认。
5. 明细：不合并相同 SKU，逐行保留原 PO 号。
6. 金额：优先复用系统已落库金额和现有金额计算口径，返利字段需在开发前确认来源。
7. 模板：新增批量 PI 模板，只保留业务确认需要的 Sheet，避免影响现有单 PO 导出。
8. 银行账户：一期明确取数规则；若账户主数据不可靠，应先给出提示或固定账户方案，避免导出错误账户。

暂不建议一期纳入：

1. 运营端同步改造。
2. 批量发送 PI 邮件。
3. 自动生成新 PI/PU 编号并回写订单。
4. 账户主数据维护流程改造。
5. 超大批量异步导出。

## 7. 初步验收标准

1. 输入多个同客户、同币种、同贸易条款 PO，可以导出一份 PI Excel。
2. 导出明细完整包含所有 PO 的 SKU 行，且每行可追溯原始小 PO 号。
3. 标准总额、调整金额、返利、运费、实际总额与系统订单数据一致。
4. 多客户、多币种、多贸易条款等异常场景有明确提示，不生成错误合并文件。
5. 批量输入能正确处理换行、空格、逗号、重复 PO、无效 PO。
6. 原有单 PO 导出功能不受影响。
