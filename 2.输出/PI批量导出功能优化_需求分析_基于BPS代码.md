# 📋 综合需求分析报告

## 1. 📝 需求概述

商城端"我的PO"页面中，跟单人员需要批量选择多个PO号（支持手动输入或粘贴多行），系统自动按同一客户维度将多个PO单的SKU商品行汇总合并，生成一份完整的PI（形式发票/Proforma Invoice）Excel文件，并支持在PI模板中补充返利(Rebate)、定金(Deposit)等金额字段，同时保留原始小PO号用于追溯。

---

## 2. 📝 需求背景与目标

### 🏢 需求背景

- 业务方（跟单团队）当前仅4名跟单人员，处理客户订单的PI制作时，需面对多PO单、品类繁杂的建店及展会类订单。客户常将一个项目的不同品类拆分为多个PO单下单，跟单员需将多个PO单的商品信息手动汇总到一份PI中，手工合并耗时长达**2-3小时**。
- 客户频繁更改拼柜方案，导致PI反复重做，极大消耗人力。随着海外门店建店、展会布置等高复杂度任务增加（物资品类涵盖软装、硬装、广告物料，SKU数量上百项），该痛点将愈发突出。

### 🎯 业务目标

- 将PI制作效率从**单次2-3小时人工操作**降低为**系统自动化生成，仅需5-10分钟核对**，释放跟单人力聚焦高价值事务。
- PI模板标准化，减少人为填写的金额计算错误和信息遗漏。

### ❗ 核心痛点

| 痛点 | 说明 |
|------|------|
| **手工合并效率低** | 同一个客户的多个PO单需手动复制粘贴合并商品行，重复劳动，耗时长 |
| **金额计算易出错** | 调价(Adjustment)、返利(Rebate)、定金(Deposit)等多维金额字段，人工计算容易出错 |
| **PI反复重做** | 客户拼柜方案频繁变更，导致PI需不断调整，缺乏系统联动 |
| **追溯困难** | 合并后的PI中SKU来自不同PO，无原始PO号标记，后续对账追溯困难 |
| **模板不完整** | 现有PI模板缺少返利(Rebate)和定金(Deposit)字段，需手动额外补充 |

## 3. 🏗️ 待澄清核心问题

### 1.[ 系统逻辑 ] 产品经理梳理（基于BPS-WS代码分析）

**(1)【数据来源】"我的PO"页面批量导出的PO数据源来自哪个系统模块？当前单个PO导出走的是paas-contract(ocContract)的数据，批量场景是否从同一数据源拉取？**<br>

- [A] 同一数据源：直接复用现有 `LsPoOrderController → PaasContractFeign` 查询链路，按PO号批量查询后合并<br>
- [B] 独立数据源：商城端前端（bff-b2b-cli）先收集PO号列表，再逐个调后端接口获取PO商品明细后合并<br>
- [C] 商城端调用运营端（bff-b2b-bus）的导出接口，由运营端统一处理数据聚合<br>

**(2)【客户校验逻辑】"仅限同一客户下多个PO单合并"的校验策略是什么？**<br>

- [A] 前端校验：用户输入PO号后，前端先调接口验证PO号是否存在且同属一个客户，不同客户则弹窗报错<br>
- [B] 后端校验：前端只传PO号列表，后端批量查询后按客户Code分组，同一客户内PO自动合并，不同客户各自生成独立Excel<br>
- [C] 前端选客户：用户先选客户，再输入该客户下的PO号，系统校验PO号归属是否匹配该客户<br>

**(3)【多客户场景】若用户输入的PO号分属多个不同客户，输出策略是？**<br>

- [A] 每个客户输出一个独立Excel文件，在同一个ZIP包中打包下载<br>
- [B] 每个客户输出一个独立Sheet页，汇总在同一个Excel文件中<br>
- [C] 直接拒绝并提示用户"请选择同一客户的PO号进行导出"<br>

**(4)【PI编号生成】会议提到"PI编号 equ PU编号"，但当前系统是否有PU编号字段？PI编号的生成规则和存储位置在哪里？**<br>

- [A] 系统已有PU概念（可在ocContract中获取PU编号），PI编号直接取PU编号字段展示到Excel中<br>
- [B] 系统当前无PU编号字段，需要在PI导出时实时生成一个新的PI编号（规则待定），并存入库表<br>
- [C] PI编号按规则（如固定前缀+年月日+序号）自动生成，无需依赖PU编号，仅用于本次导出文件命名<br>

**(5)【PO状态筛选】"我的PO"页面上PO的状态筛选逻辑是什么？导出时允许选择哪些状态的PO？**<br>

- [A] 仅允许选择"备货中"到"部分发运"阶段的PO，排除"全部发运"和"草稿审核中"的PO<br>
- [B] 允许用户自由勾选任何状态的PO，但系统自动过滤掉不适用导出的状态并在前端给出提示<br>
- [C] 增加"全部"选项，覆盖跨状态场景，允许用户自由勾选目标PO单，系统不做强制过滤<br>

**(6)【金额计算逻辑】实际总金额的计算公式，系统如何从数据库字段中取值？**<br>

- [A] 标准总金额(standardTotalAmount) - 调价(adjustment) - 返利(rebate) + 运费(freight) = 实际总金额(actualTotalAmount)，所有字段均从ocContract表中读取<br>
- [B] 标准总金额(standardTotalAmount) + 调价(adjustment) - 返利(rebate) = 实际总金额(actualTotalAmount)，运费和保费不计算在内（FOB模式）<br>
- [C] 会议提及的调价(adjustment)和discount已存在，但返利(rebate)当前在ocContract表中是否有对应字段存储，还是需要新增字段存储？需确认数据库表结构<br>

**(7)【银行账号数据源】收款银行账号信息当前在BPS客户资料库的哪个表/字段维护？不同PO间银行账号信息不一致时如何处理？**<br>

- [A] 银行账号信息存储在`ls_customer_info`或类似客户主数据表中，所有PO单取同一客户信息，不存在不一致问题<br>
- [B] 银行账号存储在合同级别（ocContract），不同PO可能绑定不同版本的信息，需取第一个PO所在合同的银行信息为准<br>
- [C] 银行账号由财务在BPS运营端维护（已有维护入口），导出时实时读取客户最新银行信息<br>

**(8)【运营端数据一致性】商城端与运营端的数据源是否完全相同？运营端增加相同导出入口时是否会有数据差异？**<br>

- [A] 完全一致：商城端(bff-b2b-cli)和运营端(bff-b2b-bus)最终都调用paas-contract的同一套数据接口，表结构和数据一致<br>
- [B] 可能不一致：两端可能使用不同的表或不同的查询逻辑，需先统一数据口径再增加导出入口<br>
- [C] 运营端有额外的价格/返利编辑功能，数据可能比商城端更完整，需评估差异后再确定同步方案<br>

### 2.[ 业务逻辑 ] 业务澄清 / 产品经理调研

**(1)【返利数据策略】PI导出中的"返利(Rebate)"字段数据来源和计算方式是什么？**<br>

- [A] 按类型汇总：系统已有多种返利类型（如门店返利、促销返利等），在PO层面已有返利金额存储，PI导出时按PO汇总各类型返利总额<br>
- [B] 人工输入：返利金额无系统数据来源，PI导出生成为空值，由跟单员在Excel中手工填写<br>
- [C] 取订单级总返利：系统记录了PO/合同级别的总返利金额，PI导出时直接取该金额，无需区分返利类型<br>

**(2)【PO号录入交互】用户在"我的PO"页面批量输入PO号，当前是已有PO列表的勾选模式，还是需要新增一个独立输入弹窗？**<br>

- [A] 现有PO列表：用户在PO列表页面已勾选多个PO后，点击"批量导出PI"按钮即可，无需新增输入框<br>
- [B] 独立输入弹窗：新增"批量导出PI"按钮，点击后弹出输入框，支持从Excel复制粘贴多行PO号（兼容回车/空格/逗号分隔）<br>
- [C] 组合模式：既支持从PO列表中勾选，也支持手动输入PO号，两种模式都支持批量生成PI<br>

**(3)【模板删减确认】"Detail调"模块确认删除后，PI模板最终呈现的结构是什么？**<br>

- [A] 仅保留"PI SH"模块，包含表头信息（客户抬头、贸易条款、卖家/买家信息、合同号）+ 商品明细行（SKU、描述、数量、单价、金额）+ 金额汇总（标准总金额、调价、返利、定金、实际总金额）<br>
- [B] 保留"PI SH"模块和简化的商品汇总，删除"Detail调"的详细分项列表，精简至一行一SKU<br>
- [C] 需进一步确认"PI SH"和"Detail调"的具体业务含义，明确删除后不会影响报关或其他下游使用<br>

**(4)【运营端入口】运营端是否需要在同一迭代中同步上线批量导出PI功能？**<br>

- [A] 同步上线：商城端和运营端同时开发批量导出PI功能，两端的PO数据源一致，同步上线后可统一使用<br>
- [B] 分阶段上线：优先上线商城端，运营端延后评估，需先验证两端数据一致性后再开发<br>
- [C] 仅在运营端上线：跟单员更习惯在后台操作，运营端才是核心使用场景，商城端PI导出后续再评估<br>

**(5)【细节处理】合并PI时，同一SKU出现在多个PO中时，是按PO分开展示为独立行，还是合计数量后展示为一行？**<br>

- [A] 按PO分开展示：每行保留原始小PO号列，同SKU不同PO视为独立行，便于追溯<br>
- [B] 合并展示：同SKU合计数量和金额展示为一行，原始小PO号在备注列中用逗号分隔列出<br>
- [C] 用户可选：提供"合并同SKU"的开关选项，让跟单员根据实际需求选择展示方式<br>

**(6)【运费和保费】FOB模式下运费和保费字段留空让手工填写，后续是否考虑支持CIF等模式的系统联动？**<br>

- [A] 当前仅满足FOB模式：运费和保费字段生成时留空，供人工填写，后续有CIF需求再评估系统联动<br>
- [B] 可扩展设计：模板中预留运费和保费字段位置，并支持从合同/发运单中读取（若有数据则填充，无数据则留空）<br>
- [C] 所有模式统一处理：不管贸易条款，运费和保费均从系统已有数据中拉取，取不到则为空<br>

## 4. 🧩 可行性初步评估

### 技术可行性：**高**

- **现有基础好**：BPS-WS已具备PO单的独立导出能力（`exportPoOrderPage`、`exportPoOrderDetail`、`exportPoTempBps`），底层的数据查询链路（bff-b2b-bus → paas-basic → paas-contract）成熟稳定。
- **改造范围明确**：主要在以下层面扩展：
  1. **前端**（商城端/运营端）：新增批量选择PO号的交互入口及输入校验
  2. **BFF层**（bff-b2b-bus/bff-b2b-cli）：新增批量PI导出的Controller接口
  3. **Service层**（paas-basic）：新增多PO数据聚合逻辑，按客户分组、SKU行合并、金额重算
  4. **模板层**：新增PI导出的Excel模板，补充返利(Rebate)、定金(Deposit)、原始小PO号等字段

### 潜在风险

1. **数据一致性风险**：商城端与运营端的数据源可能存在差异，需在上线前验证两端PO数据口径一致。
2. **返利数据来源待确认**：当前代码中虽已有返利相关的查询逻辑（`LsCustomerRebateChangeController`、`LsRebateCon`），但返利字段在PI场景中是否已有数据存储需确认。
3. **PI模板版本管理**：PI模板变更需要与业务确认正式版本（删除"Detail调"模块后是否影响其他使用场景）。

### 建议开发策略

1. **第一阶段**：优先实现商城端批量PI导出（含PO号校验、客户分组、商品行合并），验证核心链路
2. **第二阶段**：完善模板字段（返利、定金）、运营端入口、优化金额计算逻辑
